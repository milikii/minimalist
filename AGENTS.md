# AGENTS.md

本文件定义当前项目的执行规则。若与全局 AGENTS.md 冲突，以本文件为准。

---

## 项目目标

本项目采用"文档驱动 + 小步闭环 + 连续施工"的推进方式。Codex 的职责是：

- 读取项目文档
- 依据当前主线推进
- 每轮只完成一个最小闭环
- 自行验证
- 同步更新文档
- 保持默认分支长期可用

项目所有者不负责编码细节判断，因此关键改动必须可验证、可追溯、可回滚。

---

## 权威文档顺序

执行任何任务前，优先读取以下文档；若存在冲突，按顺序判定：

1. AGENTS.md
2. docs/STATUS.md
3. docs/NEXT_STEP.md
4. docs/DECISIONS.md
5. docs/ARCHITECTURE.md
6. 代码与测试真相

> 文档服务连续施工，不服务展示。若代码真相与文档冲突，先确认是否属于文档滞后；必要时修正文档。

---

## 默认执行模式

- 默认连续执行若干轮，每轮只做一个最小闭环
- 一个闭环应自包含：实施 → 验证 → 必要时更新文档 → review diff → commit
- 若仓库流程允许且远端状态明确，可 push
- 不做大杂烩式提交，不无故暂停，不频繁请求确认

---

## 当前阶段原则

除非 docs/NEXT_STEP.md 或用户明确改口，否则始终遵守：

1. 默认分支持续可用
2. 回归结果不回退
3. 当前主线持续收口
4. diff 尽量小，避免无关重构
5. 文档与代码真相保持一致

若当前阶段被定义为**质量硬化阶段**，则额外遵守：

6. 暂停新增用户可感知功能
7. 暂停扩协议、扩真相边界、扩系统能力
8. 优先降低回归风险、耦合、超大文件与历史债
9. 优先把质量 gate、focused tests、真实 smoke 补稳
10. 不用顺手加功能来掩盖结构问题

## 编码纪律

以下规则直接约束每次代码修改，优先级高于"你觉得这样更安全"的直觉。

### 错误处理：只捕获你能处理的

- 只 catch 你真正知道怎么恢复的异常。不知道怎么办 → 让它崩出去，不要吞。
- 禁止 `except Exception as e: print(e); return None` 模式，这只是在生产环境埋雷。
- 写 `try/except` 前问自己：这个 catch 让系统更正确了吗？还是只是让我不用担心了？
- 网络调用 / I/O 的合理重试除外，但要重试就正经重试（限次数、有退避），不要只 catch 一次然后返回默认值。

### 不要为"不可能发生"写兜底

- 如果 `main()` 在第 97 行注入了 `self._trace_logger`，就不要在第 140 行写 `if self._trace_logger is None: return`。
- 如果构造函数要求传 `job_repo: JobRepo`（非 Optional），就不要在方法里检查它是不是 None。
- 你写下的每个兜底都在向未来的读者暗示"这个状态是可能发生的"。如果它不可能发生，兜底就是在撒谎。
- 可选依赖（如 `feature_flag` 确实可能关闭）可以用 Optional，但 injected dependency 不是可选依赖。

### 文件边界：合并优先于拆分

- 同一个类/函数被 3+ 个消费者 import → 值得独立文件。
- 只被 1 个消费者 import → 不是模块化，是碎片，直接合并进消费者。
- 新建文件前问自己：这个文件有没有可能被另一个模块 import？如果答案是否定的，就写到调用方的文件里。
- 80 行以下的单消费者文件默认判定为"应该合并"。
- 同簇文件超过 5 个且全部是单消费者时，合并它们属于当前主线的合法工作项（即使主线没写"瘦身"）。

### 克制架构冲动

- 不要在只有 2 个状态时硬套状态机模式，if-else 够用就用 if-else。
- 不要在只有 3 步顺序逻辑时建 5 个 State 类，一个函数 + 提前 return 就是最好的状态机。
- 不要为了"未来可能扩展"预埋接口、策略模式、工厂方法，等第二次用到了再抽象。
- 如果你发现自己要新建第 6 个 `*_support.py` 文件，停下来，把它们合并成一个文件。

### 防重复

- 写任何新函数/类之前，先 grep 全仓确认没有已经存在的实现。
- 两个文件做完全相同的事、只是参数不同 → 合并，参数化。
- 发现有意义的复制粘贴代码 → 立即抽出共同部分，不要等"下次"。

### 删除比添加勇敢

- 发现死代码 → 直接删，不要注释掉，不要留 `_COMPAT_*` 元组。
- 如果某段逻辑被"以防万一"保留 → 删，git 历史记得住。
- 如果你不确定某段代码有没有用 → grep 然后判断，没找到消费者 → 删。

## 闭环选择规则

每轮只选择一个最小闭环，优先级按以下顺序判断：

1. 当前失败测试或回归红灯
2. 当前主线缺口
3. 最近引入的回归风险
4. 当前阶段明确标注的结构债
5. 可显著降低长期维护成本的最小改动

若一轮结束后没有立刻看到下一步，不允许直接停止；应重新扫描 docs/NEXT_STEP.md、docs/STATUS.md、相关代码、当前失败测试、最近若干提交，然后从"更小、更保守、可验证"的方向继续选一个闭环。

---

## 验证要求

- 能跑测试就先跑相关测试
- 有全量回归入口时，按阶段适度运行
- 涉及 downloader / import / refresh / restart / approval / persistence 等高风险链路时，优先做真实验证
- 若本机真实测试环境已就绪，不要把验证留给用户
- 若当前环境无法完成某项验证，必须明确说明缺口与影响

---

## 文档同步要求

发生以下变化时，必要文档必须同步更新：

- 当前状态变化
- 主线推进结果变化
- 质量结论变化
- 决策变化
- 架构边界变化
- 验证入口变化

> 文档写"当前真相"，不要写大段历史流水账，不要把历史台账整段抄回 STATUS.md。

---

## 质量优先原则

- 优先修真实失败与高风险耦合
- 优先降低长期维护成本
- 不为"看起来更优雅"做高风险重写
- 大文件瘦身应以边界收口、职责分离、降低协调复杂度为目标，而不是机械拆文件

---

## 日志与可追溯性要求

- 若新增或重构后台任务 / 下载 / 导入 / 刷新 / 外部服务调用 / 审批 / 重试 / 恢复 / 重启 / 跨模块编排链路，应补足最小可维护日志
- 最小要求：关键操作有结构化日志、错误日志带必要上下文、异步任务有 job/task 标识、同一业务闭环可追踪前后状态
- 不要未经需要就引入完整 observability 堆栈

---

## 模块化要求

- 业务编排、外部集成、状态协调、持久化、协议适配应尽量放在各自职责范围内
- 若存在 shared runtime / channel / service / repo 等边界，应优先消除跨层直连
- 瘦身大文件时，优先抽离稳定职责，而不是切碎调用链
- 重构必须保持行为可验证，不得凭感觉大搬家

---

## 停止条件

只有在以下情况允许停止当前连续施工：

- 出现明确 blocker
- 达到 docs/NEXT_STEP.md 当前主线的可测量退出条件
- 继续修改会违反本文件或文档边界
- 无法确认 commit / push 状态
- 已完成一个可验证闭环，且当前阶段没有比"停止并汇报"更保守的下一步

---

## 汇报条件

默认不要中途频繁汇报。只有在以下情况才汇报：失败、blocker、必须由用户决定、达到当前阶段退出条件、继续推进风险明显上升。

**全部轮次结束后**，默认只做一次简短总结，输出：

- 当前主线完成度
- 默认分支质量状态
- 本次完成的主要闭环
- 验证结果摘要
- 更新的文档
- 最后停下来的原因
- commit hash 列表（若有）

**中途异常汇报**，输出：

- 当前任务
- 当前状态
- 失败或 blocker
- 已尝试的验证
- 建议的下一步

---

## Git 纪律

- 每个闭环尽量形成独立 commit
- commit 信息应直接描述闭环内容
- 未验证通过的改动，不要伪装成完成态
- 高风险改动前，优先保留可回滚检查点
- 若仓库要求 push，先确认本地状态、分支状态与远端状态一致

## 会话落盘（codex.md）

记录最近 3 轮会话，存于仓库根目录，默认加入 .gitignore。

### 格式模板

```
## Round N

### 🎯 任务
这轮在做什么，目标是什么

### 🙋 用户
用户说了什么

### 🤖 Codex
做了什么、结论是什么、告知用户什么的核心摘要

### 🔜 下一步
下一轮做什么，或等待用户做什么决定
```

**其他规则：**

- 只保留最近 3 轮，写新轮前删最早一轮
- codex.md 默认加入 .gitignore，不提交 git
- 无法写入时必须告知用户
- 不替代 docs/STATUS.md 等项目文档
