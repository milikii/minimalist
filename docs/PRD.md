# PRD

> 2026-04-28 根据项目所有者重新确认的原始需求重写。

## 1. 产品定义

`minimalist` 的目标不是做“功能很多的代理平台”，而是做一个**稳定的 Debian NAS 宿主机旁路由控制面**。

核心交付物只有一件事：

> 在 Debian NAS 宿主机上长期稳定运行 `mihomo-core`，做 IPv4 旁路由，数月级运行尽量不出问题。

## 2. 核心用户

面向一个非常具体的用户画像：

- 自己维护 Debian NAS 宿主机
- 能接受 `systemd`、`iptables`、`ip rule`、CLI 运维
- 使用自建节点
- 希望日常只管节点和少量规则，不想维护复杂平台

## 3. 最高优先级约束

这些约束高于新增功能：

- 不能 DNS 泄露
- 默认不能影响宿主机网络
- 透明代理只服务旁路由主路径，不默认接管宿主机流量
- 配置模板必须成熟、常规、可解释
- 面板直接使用 Mihomo 推荐的现成面板，不自研 UI
- 核心主路径是“手动节点 + 自定义规则分流”

## 4. 核心范围

### 4.1 必须做好的能力

- `mihomo-core` 的安装、配置生成、systemd 启停
- 宿主机重启、服务重启后的稳定恢复
- `iptables + TProxy` 规则下发与清理
- LAN 侧 DNS 劫持，避免 DNS 泄露
- 手动节点导入、启用、禁用、改名、删除
- 少量自定义规则分流
- 基础运维检查：`status`、`healthcheck`

### 4.2 必须明确的默认姿态

- `proxy_host_output: false`
- 宿主机默认直连
- DNS 劫持只针对 LAN 入口
- 面板由外部现成方案提供

### 4.3 配置模板要求

模板应基于成熟的 Mihomo 常规思路，而不是自创结构：

- 常规 DNS 配置
- 常规 `proxy-groups`
- 常规 rules tail
- 在成熟基线之上叠加“你的个人规则分流”

也就是说，模板结构应该是：

```text
成熟常规模板
    +
你的手动节点
    +
你的自定义规则分流
```

## 5. 增强项

以下能力可以保留，但不是当前主线：

- 订阅管理
- 内核升级
- 规则仓库编辑能力
- 更复杂的 runtime audit
- cutover 工具

这些能力不得压过“长期稳定运行”的主线。

## 6. 非目标

- 自研 Web UI
- 多用户系统
- OpenWrt / nftables / dual-stack 扩边界
- 自动下载 runtime assets
- 自动升级/定时升级平台
- 把订阅能力继续扩成核心主路径

## 7. 典型使用流程

### 首次部署

1. `install-self`
2. `setup`
3. 预置 `Country.mmdb`、`GeoSite.dat`、`ui/`
4. 导入手动节点
5. 生成运行配置
6. 启动服务
7. 验证 `healthcheck` / `status`

### 日常运维

1. 增删改手动节点
2. 维护少量自定义规则
3. 重启服务
4. 用现成 Mihomo 面板观察运行状态

## 8. 成功标准

项目达到目标，不看“功能多少”，只看下面这些是否成立：

- Debian NAS 上可稳定长期运行
- LAN 流量走旁路由主路径
- DNS 不泄露
- 宿主机默认网络不被影响
- 手动节点和个人规则足够好用
- 配置模板稳定且容易理解
